# Detail View Findings

## 1. Code Block Dark Theme Bug

**Root cause**: The Shiki CSS variables (`--shiki-dark`, `--shiki-dark-bg`) are
never defined in the stylesheet. The `tiptap-styles.css` references them:

```css
html.dark .tiptap pre.shiki,
html.dark .tiptap pre.shiki code,
html.dark .tiptap pre.shiki code span {
  color: var(--shiki-dark);
  background-color: var(--shiki-dark-bg);
}
```

But these variables only exist as inline styles on individual `<pre>` elements
generated by the Shiki highlighter at runtime. The CSS variable fallback has no
value, so code blocks inherit the page background (white/transparent in dark
mode).

**Fix**: Remove the CSS variable approach and let the Shiki-generated inline
styles take precedence. Or add explicit fallback colors:

```css
html.dark .tiptap pre.shiki {
  background-color: #222222;  /* from midnight theme */
  color: #e0e0e0;
}
```

**Files**:
- `src/components/tiptap-editor.tsx` — midnight/daylight theme definitions
- `src/components/tiptap-styles.css` — lines 127-139 (the broken selectors)

## 2. Apercu Mono for Code Blocks

Font files available at `~/.local/share/fonts/apercu-mono/`:
- ApercuMonoProRegular.ttf
- ApercuMonoProMedium.ttf
- ApercuMonoProBold.ttf
- ApercuMonoProLight.ttf

**Setup**:
1. Copy font files into `src/fonts/` (gitignored)
2. Register via `next/font/local` in `layout.tsx` as `--font-code` variable
3. Apply to code blocks in `tiptap-styles.css`:
   `.tiptap pre code { font-family: var(--font-code); }`

Current font stack: Berkeley Mono (`--font-mono`) for all UI, Signifier
(`--font-serif`) for branding only.

## 3. Detail View Layout Analysis

### Current field layout (top to bottom)

```
+------------------------------------------+
| [task description input]          (title) |
+------------------------------------------+
| (time range from calendar)     (prefill) |
+------------------------------------------+
| status [dropdown]  | category [input]     |  <- flex row, 2:3 ratio
+------------------------------------------+
| due [datetime-local] | created [text]     |  <- flex row, 1:1 ratio
+------------------------------------------+
| location [input + autocomplete]           |  <- full width
+------------------------------------------+
| meeting [input + join btn]                |  <- full width
+------------------------------------------+
| [tiptap rich text editor]        (notes)  |  <- flex-1, scrollable
+------------------------------------------+
```

### The "created" stacking issue

Line 391-413 in `task-panel.tsx`: "due" (interactive datetime-local input) and
"created" (read-only text) share a flex row with `flex-1` each. Problems:

1. **Visual asymmetry**: one is an input field (h-7, border, focus ring), the
   other is plain text trying to fake the same height with `leading-7`
2. **Information hierarchy**: "created" is metadata the user rarely cares
   about — it shouldn't occupy the same visual weight as "due"
3. **Width waste**: "due" gets squeezed to 50% when "created" appears

### Layout options to consider

**Option A — Metadata footer**
Move "created" (and optionally "updated") to a muted footer bar at the very
bottom of the panel, below notes. Small text, inline:
```
created 2026-03-25 14:30 · updated 2 hours ago
```
This is how Notion handles metadata — out of the way, always visible.

**Option B — Inline after due**
Keep "created" on the same row as "due" but render it as a small muted suffix
rather than a co-equal field:
```
due [2026-03-28 09:00]  created mar 25
```
No label, just the value in `text-muted-foreground/40`.

**Option C — Collapse into hover/expand**
Only show "due" in the row. Show "created" on hover or via a small expand
chevron. Most compact but least discoverable.

**Option D — Google Calendar style**
Google Calendar's event detail shows metadata in a vertical list with icons:
```
[clock icon]  March 28, 2026  9:00 AM - 11:00 AM
[pin icon]    Clark Hall, UVA
[link icon]   https://meet.google.com/...
[calendar]    Work
```
No labels, just icon + value. "Created" doesn't appear at all in Google Calendar.

**Option E — Notion Calendar style**
Notion uses a property table: label on left (muted, small), value on right
(normal weight). Each row is its own line:
```
Status      pending
Category    Work
Due         Mar 28, 2026 9:00 AM
Location    Clark Hall
```
"Created" appears as a system property at the bottom in smaller text.

### Recommendation

**Option A (metadata footer)** is the cleanest. It frees up the "due" row to be
full-width, removes visual clutter from the editable section, and puts read-only
metadata where users expect it (at the bottom, de-emphasized). Google Calendar
doesn't show creation dates at all; Notion shows them de-emphasized at the
bottom.

### Reference: Google Calendar event detail layout

Does not repeat  (or "Every weekday")
────────────────────────────

────────────────────────────

Key takeaways:
- Date/time is prominently displayed with human-readable format, not datetime-local input
- Recurrence shown inline with date
- No "created" field
- Fields are full-width, one per row
- Icons used instead of labels
